<objective>
Wire CONC-03 curl cancellation across BOTH activity-owned curl surfaces (D-03 — RESEARCH Pitfall 1: this spans TWO distinct transports, not one) so in-flight transfers abort when the owning activity is destroyed, while happy-path transfers are unaffected (D-04: drop silently, callback returns 0 normally). Surface 1: `source/platform/mods/mod_download.cpp::download_file` already has an `xferInfo`/`CURLOPT_XFERINFOFUNCTION` — extend its `ProgressCtx` with the `cancelled` flag and abort (return 1) when set. Surface 2: `source/platform/http_client_curl.cpp::CurlHttpClient::request()` has NO progress hook today — ADD one (`CURLOPT_NOPROGRESS 0` + `CURLOPT_XFERINFOFUNCTION` + `CURLOPT_XFERINFODATA`) and extend `HttpRequest` with an optional `cancelled` field. Then thread the base-class `cancelled` flag (owned by `ThomazActivity`, Plan 01) from the activities into those calls.

Purpose: CONC-03 builds on the CONC-02 ownership model (ROADMAP planning flag "CONC-03 after CONC-02"): `ThomazActivity` owns `cancelled` and exposes it via the accessor from Plan 01; the dtor sets `*cancelled = true`. The shared-flag + XFERINFOFUNCTION design is the only viable abort path (RESEARCH "Don't Hand-Roll": Borealis async is fire-and-forget with no cancel handle). A plan touching only `mod_download.cpp` silently fails CONC-03 because cloud-save uploads and all browse GETs flow through the hook-less `http_client_curl.cpp` surface.

Output: both curl surfaces honor the cancelled flag; activities hand their base-class flag into transfers; existing non-cancelling callers unchanged.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| activity (app) → curl transport (platform) | The base-class `cancelled` flag crosses from the activity into the platform curl call path; it must be shared-owned so it survives activity destruction while curl runs on a pool thread |
| curl callback re-entrancy on teardown | `XFERINFOFUNCTION` fires on the transfer thread; reading the `cancelled` flag during/after activity teardown must be race-free |

## STRIDE Threat Register

Internal robustness refactor. No new network input is parsed; the change only ADDS an abort path. The threat class is curl-callback re-entrancy / dangling flag on teardown.

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-07 | Tampering (memory) / dangling | `cancelled` flag read by the curl callback after the owning activity is destroyed | mitigate | `cancelled` is a `shared_ptr<atomic<bool>>` captured BY VALUE into the transfer context / worker (RESEARCH Anti-Pattern: never raw pointer/reference). The flag object outlives the activity because the worker holds a `shared_ptr` copy; the base dtor sets `*cancelled = true` so the in-flight transfer aborts (callback returns 1). |
| T-04-08 | Denial of service (self) | a stuck/abandoned transfer holds a pool thread after teardown | mitigate | The XFERINFOFUNCTION abort (return 1) frees the curl `easy_perform` and the pool thread promptly on teardown instead of running to completion (RESEARCH State of the Art: "Frees pool threads on teardown"). |
| T-04-09 | Tampering | happy-path transfer wrongly aborted (regression) | mitigate | Callbacks return 1 ONLY when `cancelled && cancelled->load()`; default-null `cancelled` and the null-guard make every existing caller return 0 (unchanged). D-04: non-cancelled transfers unaffected. |
| T-04-SC | Tampering | npm/pip/cargo installs | accept | No package installs — pure in-repo refactor (RESEARCH: zero new dependencies). N/A. |

No high-severity (remotely exploitable) threat in this plan — the change only adds a local cooperative abort; stated explicitly per the security contract.
