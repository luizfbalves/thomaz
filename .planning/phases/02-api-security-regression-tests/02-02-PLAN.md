<objective>
Make access tokens revocable (SEC-02). Mint a unique `jti` on every access token in both signing sites, extend `JwtPayload` with optional `jti`/`exp`, enforce a Postgres-backed blocklist inside the existing `authenticate` decorator (jti-gated, fail-open), and add best-effort access-token revocation + a lazy expired-row sweep to `/auth/logout`.

Purpose: SEC-02 closes the HIGH-severity gap where a logged-out 365-day access token stays valid forever. The blocklist lives at the single `authenticate` enforcement point so every protected route is covered with one edit; pre-deploy tokens (no `jti`) pass unblocked (L-02), and a DB outage fails open so revocation never blocks all auth (D-06).
Output: `jti`-bearing access tokens, a `RevokedToken` blocklist check in `authenticate`, and a best-effort revoking `/auth/logout`.
</objective>

<artifacts_this_phase_produces>
This plan introduces:
- `jwtid: randomUUID()` in the `reply.jwtSign` options at BOTH access-token signing sites: `signAuthResponse` (api/src/lib/auth-tokens.ts) and the inline `/auth/refresh` handler (api/src/routes/auth.ts).
- `jti?: string;` and `exp?: number;` added to the `JwtPayload` type (api/src/plugins/auth.ts) — optional because pre-deploy tokens lack them; the `@fastify/jwt` module augmentation reuses this type so one edit covers `request.user` and `jwtVerify` generics.
- A blocklist check inside the `authenticate` decorator: after `jwtVerify()`, skip when no `jti` (D-05/L-02), else `revokedToken.findUnique({ where: { jti } })`; revoked → generic 401; DB error → warn + allow (D-06).
- Best-effort access-token revocation in `/auth/logout`: best-effort `jwtVerify`, lazy `deleteMany({ where: { expiresAt: { lt: now } } })` (D-09), then `upsert` of the current `jti` with `expiresAt = new Date(decoded.exp * 1000)` (D-08, Pitfall 3 upsert).
</artifacts_this_phase_produces>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → /auth/logout | Untrusted (possibly absent/expired) bearer; logout must stay open and best-effort |
| client → protected routes | Bearer crosses into `authenticate`; revoked tokens must be rejected |
| API → Postgres (RevokedToken) | Blocklist lookup/insert; a DB outage must not block all auth |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-04 | Spoofing | Replay of a logged-out 365-day access token | mitigate | `jti` blocklist enforced in `authenticate`; revoked `jti` → generic 401 (SEC-02). Pre-deploy tokens without `jti` age out via relogin (L-02) — accepted, locked behavior |
| T-02-05 | Information Disclosure | Distinct revocation error code leaking token state | mitigate | Return the generic `{ ok:false, error:"unauthorized" }` (D-03); no `token_revoked` code — no oracle distinguishing revoked vs expired |
| T-02-06 | Denial of Service | Revocation DB outage blocking all authenticated traffic | mitigate | Fail-open: `findUnique` wrapped in try/catch, warn + allow on error (D-06) — availability prioritized over strict revocation |
| T-02-07 | Denial of Service | `/auth/logout` abused against the rate-limited /auth/* group, or double-logout P2002 crash | mitigate | Logout stays best-effort and always-200; no authenticate preHandler and no authRateLimit added (Pitfall 4); `upsert` survives double-logout (Pitfall 3); a missing/invalid bearer cannot 401 (no new auth-bypass/DoS surface) |
| T-02-08 | Denial of Service (storage) | Unbounded RevokedToken growth | mitigate | Lazy `deleteMany({ expiresAt: { lt: now } })` sweep on each logout (D-09) keeps the table bounded without a scheduler |
</threat_model>
