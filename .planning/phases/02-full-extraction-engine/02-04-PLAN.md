<objective>
Verify the full extraction engine on real hardware: run extract_all_base_layouts()
under title takeover and confirm the canonical layouts land in the flat
/themes/systemData/ layout. This is the phase gate for EXTRACT-01/02/03 success
criteria 1-3, which the research confirms are hardware-only and not automatable
(the privileged BIS/SPL path does not run on the desktop host build).

This plan introduces NO code changes — Plans 01-03 deliver all source. It is a
single human-verify checkpoint that drives the on-device run and inspects the SD
output.

Purpose: prove the engine works end-to-end on the target firmware before Phase 3
wires it into the theme UI.
Output: confirmed SD-card output set + a verification record for the SUMMARY.
</objective>

<task type="checkpoint:human-verify" gate="blocking">
  <name>Task 2: Hardware verification — run extract_all_base_layouts() under title takeover</name>
  <what-built>
    extract_all_base_layouts() — the Phase 2 multi-title extraction engine. It opens
    the privileged session once, iterates qlaunch (0100000000001000), Psl
    (0100000000001007), and MyPage (0100000000001013), captures every /lyt/*.szs,
    structurally validates each (Yaz0+SARC), and writes the basename flat into
    /themes/systemData/ with overwrite. Plus the /lyt/ prefix RomFS filter and the
    structural validator.
  </what-built>
  <how-to-verify>
    1. Build the Switch target (CI image devkitpro/devkita64, or the project's normal
       NRO build) and copy the NRO to the SD card.
    2. Launch thomaz via title takeover (hold R while opening a game) so it runs in
       Application mode — the same launch path documented in Phase 1
       (docs/title-takeover.md).
    3. Trigger extract_all_base_layouts() via the call site documented in Task 1.
    4. Inspect /themes/systemData/ on the SD card. CONFIRM all six qlaunch layouts are
       present as flat files: ResidentMenu.szs, Entrance.szs, Flaunch.szs, Set.szs,
       Notification.szs, common.szs (success criterion 1).
    5. CONFIRM Psl.szs and MyPage.szs are also present (success criterion 2). With the
       /lyt/ prefix filter (D-01) additional layouts (e.g. Option.szs, Eula.szs) may
       also appear — that is expected and fine.
    6. CONFIRM every file is DIRECTLY in /themes/systemData/ — a flat layout, NOT
       nested under extracted/qlaunch/ or any subdir (success criterion 3).
    7. CONFIRM each .szs is non-empty and (if you have host tooling) Yaz0/SARC-valid;
       the engine already structurally validated them before writing.
    8. Read the run log: note written_parts count and any failed_parts. A missing
       optional part reported in failed_parts WITHOUT aborting the run is the correct
       D-02 best-effort behavior (success: ok=true even with a few failed_parts).
    9. (Negative path, optional) Launch in applet mode (normal, no title takeover) and
       trigger extraction; CONFIRM it returns the "Relaunch via title takeover" message
       and writes nothing — no crash (TAKEOVER-01 carried forward).
    10. Record the firmware version printed by the run (setsysGetFirmwareVersion) for
        the SUMMARY provenance.
  </how-to-verify>
  <resume-signal>
    Type "approved" if all six qlaunch layouts + Psl + MyPage are present, flat, and
    non-empty in /themes/systemData/ and the run did not abort on a missing optional
    part. Otherwise describe what was missing/wrong (e.g. which szs absent, any
    crash, any nesting, any systemic abort) so the driver can be corrected.
  </resume-signal>
</task>
