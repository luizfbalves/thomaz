<objective>
Extract the D-04 structural szs validation into a neutral, host-testable
translation unit so success criterion 4 (validation covered by host doctest where
it does not touch privileged services) is met.

Phase 1 shipped `is_valid_szs` (a 4-byte magic check) inside the Switch-only
firmware_extract_switch.cpp; that check passes truncated/corrupt buffers
(Pitfall 4). D-04 requires the stronger check: a buffer is valid only if it
Yaz0-decompresses (when Yaz0-compressed) and unpacks as a non-empty SARC. SarcLib
(`Yaz0::Decompress` / `SARC::Unpack`) already implements both and touches no
privileged services — so the validator belongs in a neutral TU that links into
BOTH the Switch app (C++20, SarcLib already linked) and a host doctest.

This plan creates `szs_validate.{hpp,cpp}` + a host test, and bumps the tests
Makefile to C++20 (backward compatible — see CMakeLists.txt:125-127) with SarcLib
sources added so the host test can link the real Yaz0/SARC implementation.

Purpose: turn the Phase 1 magic-only check into D-04's structural check AND make
it host-testable (success criterion 4 for validation). Plan 03's driver consumes
this validator before every write.
Output: neutral szs_validate TU, host doctest, C++20 tests Makefile wiring.
</objective>
