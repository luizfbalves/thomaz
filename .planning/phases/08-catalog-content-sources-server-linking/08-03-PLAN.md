<objective>
Add the cloud half of server-linking: exactly one owner-scoped Prisma model (`SourceLink`) and
one JWT-authed Fastify route (`/sources`) that stores server-link CONFIG ONLY — never catalog or
content bytes — with the credential encrypted at rest, plus the device-side
`http_source_sync_client` that mirrors the cloud-saves client. Backed by a Vitest spec proving
owner-scoping, config-only, and at-rest encryption.

Purpose: SYNC-01 (one-tap config sync, config only, reusing JWT + the cloud-saves pattern) and
SYNC-02 (credentials protected at rest, scoped to the owning account). This is a near-mechanical
clone of the `SaveSlot` + `saves.ts` + `HttpCloudSaveClient` trio with the deliberate ABSENCE of
any blob/multipart handling as the structural proof that it is config-only.

Output: `SourceLink` model + migration, `source-crypto.ts`, `sources.ts` route registered in
`app.ts`, a `sources.test.ts` Vitest spec, and the device `http_source_sync_client`.
</objective>

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-08-09 | Information Disclosure | credentials readable at rest in the cloud DB | mitigate | `authSecretEnc` stored AES-256-GCM encrypted via `source-crypto.ts`; DTO returns `hasSecret`, never the secret; Vitest asserts stored column != plaintext (V6). |
| T-08-10 | Information Disclosure | credential leaking in API logs | mitigate | Existing pino redaction of `authorization`/`cookie` (`app.ts`); the secret is never placed in a header/URL the logger captures; device client never logs the body (Pitfall 7). |

| T-08-12 | Spoofing | unauthenticated access to /sources | mitigate | `preHandler:[app.authenticate]` on every route (existing JWT verify + revocation, V2/V3). |
</threat_model>
