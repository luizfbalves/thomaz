<objective>
Prevent repeated button taps from queueing duplicate downloads or applies during
an in-flight async operation on the theme-detail screen.

Purpose: The download path has no busy guard — a fast double-tap during a download
launches two concurrent download_theme() calls. Apply and remove already guard via
`this->busy` but do not give visual feedback (button stays lit and focusable).

Output: A private `setButtonBusy(bool)` helper that (a) sets/clears `this->busy`,
(b) toggles `setFocusable` + `setAlpha` on the `downloadButton` view, and (c) is
called consistently from startDownload, doApplyMode, and doRemove — plus a single
`if (this->busy) return;` guard at the top of the click handler.
</objective>

    void ThemeDetailActivity::setButtonBusy(bool busy) {
        this->busy = busy;
        auto* btn = this->getView("downloadButton");
        if (!btn) return;
        btn->setFocusable(!busy);
        btn->setAlpha(busy ? 0.5f : 1.0f);
    }

Then do a targeted find-and-replace across the three async methods:

1. startDownload(): at the top of the function body, after the `if (!this->resolved) return;` guard, add:
       if (this->busy) return;
       this->setButtonBusy(true);
   In the onSync lambda, replace the existing `brls::Application::notify(results->second);` block so that
   BOTH the failure path and the success path call `this->setButtonBusy(false)` before any other state
   update. The completion block should become:
       this->setButtonBusy(false);
       brls::Application::notify(results->second);
       if (results->first) {
           this->downloaded = true;
           this->refreshActionButton();
           this->analyzeCompat();
       }

2. doApplyMode(): replace `this->busy = true;` with `this->setButtonBusy(true);` and replace
   `this->busy = false;` (the reset in onSync) with `this->setButtonBusy(false);`. There is exactly one
   of each in this function — do not alter any other logic.

3. doRemove(): same substitution — replace `this->busy = true;` and `this->busy = false;` with the
   setButtonBusy calls. One of each, in the same positions.

4. In the click handler registered in onContentAvailable (the lambda that calls startDownload /
   doApply / doRemove), add a busy-gate as the very first check inside the brls::sync lambda,
   immediately after the alive guard:
       if (this->busy) return;
   This is a belt-and-suspenders guard: setButtonBusy already makes the button non-focusable, but
   gesture recognizers and programmatic calls can bypass focus — the early return closes that gap.

Do NOT touch doApply() (the routing function — it already has `if (!this->resolved || this->busy)

and uses brls::async not runAsync — leave it unchanged). Do NOT change any dialog callbacks or the
analyzeCompat/updateCompatBadge flow.
