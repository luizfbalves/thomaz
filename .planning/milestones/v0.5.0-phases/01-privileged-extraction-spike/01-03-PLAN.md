<objective>
Port a minimal hactool wrapper: a thin C++ facade over the vendored hactool C fork that decrypts an
NCA's RomFS in-memory using the SPL-derived header key, applies a `/lyt/*.szs` filename filter, and
returns the matched files as byte buffers (no disk dump of the whole RomFS). For the spike the filter
list is the single entry `/lyt/ResidentMenu.szs` (D-02). This is real keeper code (D-01) that plan 04
composes with the key_loader (plan 02), and Phase 2 extends to more files.

Purpose: this is the EXTRACT-04 decrypt step — using the on-device SPL-derived header key (NOT a
`prod.keys`) loaded into the hactool keyset to decrypt and read the RomFS. The facade confines ALL
hactool C-API knowledge to one file (mirroring how `apply_facade.cpp` confines the theme engine), so
the rest of thomaz never sees `nca_ctx_t`.

Output: `nca_extract_switch.{hpp,cpp}` — a platform-neutral facade header (no hactool/libnx types in
the signature) plus the `__SWITCH__`-guarded implementation. This plan does NOT mount BIS / derive the
key (plan 02) and does NOT add the entry point / applet gate / SD write (plan 04).
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| encrypted firmware NCA → in-memory decrypt | Untrusted/copyrighted firmware bytes are decrypted in process memory; only the user's SD is a valid sink (write happens in plan 04). |
| hactool C API → thomaz | All `nca_ctx_t`/keyset knowledge is confined to this one facade file (no leakage into the rest of thomaz). |
| SPL-derived header key → hactool keyset | The sensitive key is copied into the keyset in memory only. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-09 | Tampering / Integrity | hactool decrypt output | mitigate | On bad decrypt / empty result, return an empty map + error (never a partial/garbage buffer); the caller (plan 04) validates szs magic before writing. |
| T-01-10 | Information Disclosure | Header key in keyset | mitigate | Header key lives only in the in-memory hactool keyset for the duration of the call; never logged, never persisted (EXTRACT-04). |
| T-01-11 | Information Disclosure / Copyright | Decrypted RomFS bytes | mitigate | Decrypted szs bytes stay in memory and (plan 04) are written ONLY to the user's local SD; never uploaded or distributed. |
| T-01-12 | Tampering | AES-CMAC primitive | mitigate | Use `MBEDTLS_CMAC_C` from the plan-01 CMAC build (never hand-roll CMAC — ASVS V6). |
| T-01-13 | Denial of Service | Malformed NCA crashes the fork | mitigate | Check the fork's error returns; surface a clean error string rather than crashing (the fork builds with `-Wall`; relax `-Werror` only on the vendored target per A2, not on error handling). |
</threat_model>
