<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| async pool thread → UI thread | Worker runs while the activity may be popped; the (now base-owned) `alive` guard mediates the `onSync` continuation |
| view-id string → typed brls::View* | `getView(id)` returns a base `View*`; a mistyped id / wrong-type view is an untrusted cast that must fail safe |

## STRIDE Threat Register

Internal robustness refactor; no network/auth/input-validation surface changes. The threat class IS the subject (use-after-free on guard removal; bad-pointer crash on view cast).

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-03 | Tampering (memory) / use-after-free | removed per-activity `alive` member leaves a dangling guard reference | mitigate | The base `ThomazActivity` owns `alive`; derived activities reference the inherited member; `runAsync` captures it by value. Each migration removes BOTH the member and its dtor line in the same pass (RESEARCH Pitfall 5) so no `'alive' is not a member` dangling state ships. Verified by clean desktop build. |
| T-04-04 | Denial of service (crash) | C-style cast on a wrong-type/missing view dereferenced later | mitigate | `dynamic_cast<brls::T*>` + `if (!x) { log; return; }` — a mistyped/absent view id fails safe (logged no-op) instead of a null/bad-pointer dereference crash (DEBT-03 / ROADMAP success criterion 2). |
| T-04-SC | Tampering | npm/pip/cargo installs | accept | No package installs — pure in-repo refactor (RESEARCH: zero new dependencies). N/A. |

No high-severity (remotely exploitable) threat in this plan; stated explicitly per the security contract.
</threat_model>
