# Spike: Cheat File Format + switch-cheats-db Mapping

**Date:** 2026-06-02
**Status:** Complete
**Verdict:** **NUANCED GO.** The db exposes the mapping directly: `versions/<TITLE_ID>.json` is keyed by the raw `u32` version value returned by `nsListApplicationContentMetaStatus` (no conversion needed) and maps to a `build_id`. Reliable for titles with cheat coverage. Two fallbacks are required: (a) when the version key is absent, read the build_id from the local NSO; (b) when the mapped build_id has **no cheats yet** (db coverage lags new game versions), degrade gracefully to the most recent build_id that *does* have cheats. See §3.4. Details below.

## 1. Cheat File Format

### 1.1 Load Path

Source: [Atmosphère cheats.md](https://github.com/Atmosphere-NX/Atmosphere/blob/master/docs/features/cheats.md)

```
/atmosphere/contents/<program_id>/cheats/<build_id>.txt
```

- `program_id` — the 64-bit title ID in lowercase hex (16 hex digits), e.g. `0100000000010000`
- `build_id` — **first 8 bytes of the application's main NSO executable's build ID**, rendered as 16 uppercase hex digits, e.g. `B424BE150A8E7D78`

Exact quote from the Atmosphère docs:
> "where `build_id` is the hexadecimal representation of the first 8 bytes of the application's main executable's build id."

### 1.2 Text File Structure

The `.txt` file is plain text. Each cheat entry has:

1. **Section header** — either `[Cheat Name]` (regular cheat) or `{Cheat Name}` (master/anchor code). The header line is the cheat's display name.
2. **Opcode lines** — one or more 8-hex-digit words per line, space-separated.
3. **Blank line** — separates entries (optional but conventional).

No dedicated comment syntax is documented in the official Atmosphère docs. The GBAtemp community files use `#` as a comment prefix in some attribution `.txt` files, but not in the cheat opcode files themselves.

**Master / anchor code:** A section whose header is wrapped in braces `{Name}` instead of square brackets. By convention it is named `{master}` but the braces syntax is what the parser matches. The master code is applied unconditionally before any toggled cheats; it typically sets up pointer chains or conditions the other cheats depend on.

Source: `process_cheats.py` from switch-cheats-db, line:
```python
titles = re.search(r"(\[.+\]|\{.+\})", lines[i])
```
This regex is the canonical parser: any line matching `[...]` or `{...}` is a section header; everything after it (until the next header) is that cheat's opcodes.

**Opcode line format:** Space-separated 8-character uppercase hex words. The high nybble of the first word is the instruction type (e.g. `0` = write static value, `1` = begin conditional, `2` = end conditional, `8` = keypress conditional, `5`/`7`/`6` = register/memory ops). Full encoding details: [Atmosphère cheats.md §Cheat Code Format](https://github.com/Atmosphere-NX/Atmosphere/blob/master/docs/features/cheats.md#cheat-code-format).

### 1.4 Parser Rules for thomaz

To split a `.txt` file into named cheats:

1. Read lines sequentially.
2. When a line matches `^\[.+\]$` → start a new regular cheat; the line content (including brackets) is the name.
3. When a line matches `^\{.+\}$` → start a master/anchor cheat; same name convention.
4. All non-header, non-blank lines following a header are that cheat's opcodes.
5. A blank line between entries is cosmetic; it does not end an entry — only the next header ends it.

To re-serialize "only enabled cheats + master code":

```
output = ""
for each master cheat (headers in braces):
    output += header_line + "\n"
    output += opcode_lines + "\n\n"
for each enabled regular cheat:
    output += header_line + "\n"
    output += opcode_lines + "\n\n"
```

By default Atmosphère **enables all cheats** in the file on load. To disable specific cheats at boot, the cheat manager service API must be used at runtime, OR the `.txt` file itself should only include the cheats the user wants active (which is thomaz's approach).

## 2. switch-cheats-db Layout

Source: [HamletDuFromage/switch-cheats-db](https://github.com/HamletDuFromage/switch-cheats-db)

### 2.1 Repository Folder Layout

```
switch-cheats-db/
├── cheats/                  # Main cheat JSON store (Atmosphère + GBAtemp combined)
│   └── <TITLE_ID>.json      # One file per title, e.g. 0100000000010000.json
├── cheats_gbatemp/          # GBAtemp-only cheats (same JSON format)
│   └── <TITLE_ID>.json
├── cheats_gfx/              # 60fps/resolution/graphics cheats only
│   └── <TITLE_ID>.json
├── versions/                # Per-title version→build_id mappings
│   └── <TITLE_ID>.json      # e.g. 01006A800016E000.json (Super Smash Bros. Ultimate)
├── versions.json            # Master mapping: all titles in one file (3.32 MB)
├── process_versions.py      # Script that builds versions/ and versions.json
├── process_cheats.py        # Script that parses upstream .txt → JSON
├── database_builder.py      # Orchestrator: downloads, parses, packages releases
└── README.md
```

**Important:** The `cheats/` directory stores cheats as **JSON** (not raw `.txt`). Each JSON key at the top level is a `build_id` (16-char uppercase hex). The value is a dict of `"[Cheat Name]" → "full cheat text including header\n"`.

The `.txt` files only exist inside the **release zip assets**. They are generated on-the-fly by `database_builder.py`'s `build_cheat_files()` function.

## 3. Version → Build ID Mapping (The Risk Assessment)

### 3.1 What the `ns` service returns

Source: libnx `nx/include/switch/services/ns.h`
URL: `https://github.com/switchbrew/libnx/blob/master/nx/include/switch/services/ns.h`

The relevant struct:
```c
typedef struct {
    u8  meta_type;      // NcmContentMetaType
    u8  storageID;      // NcmStorageId
    u8  rights_check;   // RightsCheck
    u8  reserved;
    u32 version;        // Application version  <--- this is the key
    u64 application_id; // ApplicationId (= title_id)
} NsApplicationContentMetaStatus;
```

`nsListApplicationContentMetaStatus(application_id, index, list, count, &out_entrycount)` returns one entry per installed content metadata record (base game, update, DLC). The `version` field is the raw `u32` display version integer.

### 3.2 The Mapping Format

The `versions/<TITLE_ID>.json` files use the **stringified `u32` version integer as the key**. This is the **same value** returned by `NsApplicationContentMetaStatus.version`. No conversion is needed.

**Concrete traced example — Super Smash Bros. Ultimate:**
- Title ID: `01006A800016E000`
- `nsListApplicationContentMetaStatus` for this game with the latest update installed returns `version = 1966080`
- Look up `versions/01006A800016E000.json`:
  ```json
  {
      "0": "FD471EB458C915E7",
      "524288": "343D891A4BCF0684",
      "1048576": "D45361E905C9E3C1",
      ...
      "1966080": "3EAE0063B12FD81E",
      "latest": 1966080,
      "title": "Super Smash Bros.™ Ultimate"
  }
  ```
- `str(1966080)` → key `"1966080"` → build_id `"3EAE0063B12FD81E"`
- Cheat file URL in the raw repo:
  ```
  https://raw.githubusercontent.com/HamletDuFromage/switch-cheats-db/master/cheats/01006A800016E000.json
  ```
  ⚠️ **Verified caveat (coverage lag).** This title's *latest* build_id `3EAE0063B12FD81E` is **not** present in the cheats file — `cheats/01006A800016E000.json` contains only older build_ids (most recent with cheats observed: `B9B166DF1DB90BAF` = version `1769472`). So the version→build_id lookup *succeeds*, but **no cheats exist for that exact build_id**. This is the real edge case Phase 2 must handle gracefully (see §3.4 fallback b). The Super Mario Odyssey trace below is fully end-to-end — cheats exist for its latest build_id.
- On SD card: `/atmosphere/contents/01006a800016e000/cheats/<build_id>.txt`

**Second trace — Super Mario Odyssey:**
- Title ID: `0100000000010000`
- Latest version `u32` = `393216`
- `versions/0100000000010000.json` → key `"393216"` → build_id `"B424BE150A8E7D78"`
- `/atmosphere/contents/0100000000010000/cheats/B424BE150A8E7D78.txt`

Version number encoding (confirmed via Python decode): the u32 encodes `major` in bits 17+ (i.e. `major = version >> 17`), so v1.0.0 = `131072`, v4.0.0 = `524288`, v6.0.0 = `393216` (for 16-bit aligned titles that use a different packing). The exact encoding does not matter to thomaz — the raw `u32` value is the lookup key.

### 3.3 Coverage Assessment

Source: `versions.json` (3.32 MB, all titles) and the `versions/` directory (19,583+ per-title files observed in the tree listing).

The version mapping is populated by `process_versions.py`, which pulls from Nintendo's own content metadata endpoints (via the Eshop CDN and libnx-compatible data sources). Coverage is not 100%: the db has ~2,501 game titles with cheats, but version metadata may lag behind for brand-new releases. For any title in the db, the version→build_id mapping is available if the title received cheat coverage.

**Gap:** If a game has a cheat entry in `cheats/<TITLE_ID>.json` but no corresponding entry in `versions/<TITLE_ID>.json`, the version-based lookup fails and the app must fall back to reading the build_id locally from the installed NSO. This is an edge case, not the common path.

### 3.4 Verdict

**NUANCED GO** (with a mandatory fallback path identified):

The strategy of using `ns` version data + the switch-cheats-db `versions/` index is **viable and reliable** for the common case. The mapping is a direct `u32 → build_id_string` lookup with no transformation required. For titles where the version file exists, the approach works without touching the NSO binary at all.

However, the following fallbacks must be implemented in Phase 2:

- **(a) Version key absent.** If `versions/<TITLE_ID>.json` does not exist OR does not contain the specific version key, read the build_id locally from the installed NSO via the `ro` service (or by reading the NSO header from the installed content path). Triggered only when the db mapping is absent.
- **(b) build_id mapped but no cheats (coverage lag).** The version map may resolve to a build_id (often the `latest`) for which the cheats file has **no entries yet** — verified live for Super Smash Bros. Ultimate (§3.2). In this case, degrade gracefully: pick the most recent build_id present in `cheats/<TITLE_ID>.json` and surface it to the user clearly (e.g. "cheats available for an older version"). Showing older cheats beats showing nothing; never write a cheat file for a build_id that has no cheats.

The risk originally identified ("go/no-go") is resolved: **GO** on the version-index strategy as primary, with two well-defined fallbacks.

## 4. Phase 2 Inputs

These are the concrete constants Phase 2 code will hardcode or use at runtime. No guessing required.

### Cheat Load Path Template (on SD card)
```
/atmosphere/contents/{title_id_lower}/cheats/{build_id_upper}.txt
```
- `title_id_lower`: 16-char lowercase hex, e.g. `0100000000010000`
- `build_id_upper`: 16-char uppercase hex first-8-bytes of NSO build ID, e.g. `B424BE150A8E7D78`

### libnx Call to Get Installed Version
```c
NsApplicationContentMetaStatus status[16];
s32 count = 0;
nsListApplicationContentMetaStatus(application_id, 0, status, 16, &count);
u32 version = status[0].version; // raw u32 — direct lookup key
```

### switch-cheats-db Base URL
```
https://raw.githubusercontent.com/HamletDuFromage/switch-cheats-db/master
```

### Version Mapping Fetch URL
```
{base_url}/versions/{TITLE_ID_UPPER}.json
```
Example:
```
https://raw.githubusercontent.com/HamletDuFromage/switch-cheats-db/master/versions/0100000000010000.json
```

### Cheat Data Fetch URL (per-title JSON, no full archive download needed)
```
{base_url}/cheats/{TITLE_ID_UPPER}.json
```
Example:
```
https://raw.githubusercontent.com/HamletDuFromage/switch-cheats-db/master/cheats/0100000000010000.json
```

### Version→Build ID Lookup Algorithm
```python
# Pseudocode
cheats = fetch_json(f"{BASE_URL}/cheats/{title_id.upper()}.json")   # keys = build_ids with cheats
version_data = fetch_json(f"{BASE_URL}/versions/{title_id.upper()}.json")
build_id = version_data.get(str(version_u32))   # version_u32 is NsApplicationContentMetaStatus.version
if build_id is None:
    build_id = read_build_id_from_local_nso(title_id)        # fallback (a): version key absent
if build_id not in cheats:
    build_id = most_recent_build_id_with_cheats(cheats)      # fallback (b): coverage lag, degrade gracefully
    # surface to user: "cheats available for an older version"
# never write a cheat file for a build_id that has no cheats
```

### Cheat JSON Structure (for the parser)
```
{
  "<BUILD_ID_16_HEX>": {
    "[Cheat Name]": "[Cheat Name]\nOPCODE1\nOPCODE2\n\n",
    "{Master Code}": "{Master Code}\nOPCODE\n\n",
    ...
  },
  "attribution": { ... }  // ignore for cheat display
}
```

### Release Zip for Full Offline Install
```
https://github.com/HamletDuFromage/switch-cheats-db/releases/latest/download/contents_complete.zip
```
Internal path: `contents/<title_id>/cheats/<build_id>.txt`

### VERSION File (for staleness check)
```
https://github.com/HamletDuFromage/switch-cheats-db/releases/latest/download/VERSION
```
